# Acceptable Data Race in HeavyKeeper Fading

`HeavyKeeper.fading()` performs non-atomic `long[] >>= 1` without the per-stripe lock that `add()` uses, risking a torn `long` read on 32-bit JVMs. This is a deliberate trade-off — locking `fading()` would add contention on a path that runs every 30s, and switching to `AtomicLongArray` would bloat every counter and degrade `add()` throughput. HeavyKeeper is a Count-Min Sketch variant; a lost increment on a single decay cycle is at most one count lost per 30s per counter, which negligibly delays hot promotion by one cycle for a boundary key. Keep `>>=1` unlocked. Do not change to `AtomicLongArray`. Do not lock `fading()`.
