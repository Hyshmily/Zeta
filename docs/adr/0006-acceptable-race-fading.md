# Per-Stripe Lock in HeavyKeeper Fading

`HeavyKeeper.fading()` halves all sketch counters under per-stripe locks (`synchronized (lockStripes[i & lockMask])`) to prevent concurrent `addDirect()` from observing torn `long` values (JLS 17.17). The original design deliberately left `fading()` unlocked as a performance trade-off, arguing that a single lost increment per 30s decay cycle was acceptable. However, on 32-bit JVMs the torn read can produce values that are neither the old nor the new value — a corrupted intermediate that corrupts all subsequent decay cycles for that bucket until a zero-write resets it. The per-stripe lock adds negligible contention because `fading()` runs once per decay interval (~30s) while `addDirect()` holds the same stripe lock for a single `long` write (nanoseconds). We explicitly chose NOT to use `AtomicLongArray` because it would bloat every counter (object header per element) and degrade `add()` cache locality. Lock the decay, not the counters.
