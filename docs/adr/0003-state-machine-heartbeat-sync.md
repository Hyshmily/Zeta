# State Machine Config via Heartbeat Gossip

Zeta's `ZetaBayesianSM` parameters (confirmCount, coolCount, preCoolGraceCount) are synchronized across Worker instances via AMQP heartbeat broadcasts — not Redis, not etcd, not a separate control bus. Each heartbeat carries a `configTimestamp` header; peers apply it when strictly newer. New Workers wait ≤3s on startup for a peer heartbeat before falling back to `WorkerProperties` defaults. This decentralized gossip is justified by the target deployment model (rolling/k8s, never all Workers down simultaneously), avoiding an external config store dependency. The `StateMachineEndpoint` (GET/POST `/actuator/hotkey/worker/state`) was added to the common module rather than the worker module so the actuator path is available regardless of which module is deployed.
