# Dual Version Space: dataVersion vs decisionVersion

HotKey tracks two independent version spaces to decouple data mutations from Worker decisions. `dataVersion` uses Redis `INCR` with local `Long.MIN_VALUE + counter` fallback when Redis is unavailable (marked `isVersionDegraded=true`), and peers apply a 4-case comparison to prevent degraded versions from overwriting normal ones. `decisionVersion` uses a Worker-local `AtomicLong` that never degrades — if the Worker restarts the counter resets to 0, and `shouldSkipForWorker()` unconditionally accepts incoming decisions when the existing entry is degraded. This guarantees a Redis outage never blocks HOT/COOL delivery, and a write storm never disrupts decision lifecycles. The cost is two version fields in every `CacheEntry` (24 bytes overhead) and `VersionGuard` to centralise comparison logic.
